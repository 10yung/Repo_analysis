[Compare Source](https://togithub.com/styled-components/styled-components/compare/v4.4.1...v5.0.0)

It's finally here!!! ðŸš€See the [migrating to v5 FAQ page](https://www.styled-components.com/docs/faqs#what-do-i-need-to-do-to-migrate-to-v5) for easy upgrade instructions!

### Read the [v5 release announcement](https://medium.com/styled-components/announcing-styled-components-v5-beast-mode-389747abd987)!

#### TL;DR:

-   19% smaller bundle size
-   18% faster client-side mounting
-   17% faster updating of dynamic styles
-   45% faster server-side rendering
-   RTL support

...and much more all, with **no breaking changes!**

**Changes that happened since the initial announcement**

-   `StyleSheetManager` enhancements

    -   you can now supply stylis plugins like [stylis-plugin-rtl](https://www.npmjs.com/package/stylis-plugin-rtl); `<StyleSheetManager stylisPlugins={[]}>...</StyleSheetManager>`
    -   `disableVendorPrefixes` removes autoprefixing if you don't need legacy browser support; `<StyleSheetManager disableVendorPrefixes>...</StyleSheetManager>`
    -   `disableCSSOMInjection` forces using the slower injection mode if other integrations in your runtime environment can't parse CSSOM-injected styles; `<StyleSheetManager disableCSSOMInjection>...</StyleSheetManager>`

-   Removed the "subfunction" attrs syntax that was deprecated in v4

    ```js
    styled.div.attrs({ role: p => p.onClick ? 'button' : '' })`
      color: red;
    `
    ```

    becomes

    ```js
    styled.div.attrs(p => ({ role: p.onClick ? 'button' : '' }))`
      color: red;
    `
    ```

-   Update css-to-react-native to v3.0.0 ([#&#8203;2811](https://togithub.com/styled-components/styled-components/issues/2811)); the one breaking change noted is that unitless line height is no longer allowed when setting font properties

    -   if you get a lot of warnings at dev time from not having units in your styles, this codemod is available to help expedite updates: <https://github.com/styled-components/styled-components-native-code-mod> thanks [@&#8203;jacobp100](https://togithub.com/jacobp100)

-   disallow /ad/i in generated class names ([#&#8203;2837](https://togithub.com/styled-components/styled-components/issues/2837)); this change primarily helps to avoid some overly aggressive ad blockers that will mangle generated classnames containing the substring "ad"

**NOTE: At this time we recommend not using `@import` inside of `createGlobalStyle`. We're working on better behavior for this functionality but it just doesn't really work at the moment and it's better if you just embed these imports in your HTML index file, etc.**

Tracking issue for making the documentation more relevant. 

We've seen a few issues (#980, #1002) that appear to be related to brand new projects attempting to use ES module imports. However, the getting started pages ([here](https://reactjs.net/getting-started/aspnetcore.html), [here](https://reactjs.net/tutorials/aspnetcore.html), and [here](https://reactjs.net/getting-started/usage.html)) do not set new projects up with a compatible configuration.

My recommendation: The happy path for getting new projects started with this library should recommend using a bundler, with as minimal of a configuration as possible to support. Optionally we can link out to sites like [createapp.dev](https://createapp.dev/webpack) to preempt questions about more complex configurations. A few bundlers to investigate as the recommended starting point:

- [create-react-app](https://reactjs.org/docs/create-a-new-react-app.html), which ships its own webpack configuration. This project supports very limited customization, asking devs to either "eject" or to maintain a fork of the scripts repo
- [parcel](https://parceljs.org/recipes.html#react), which has built in defaults for React projects, and can be extended
- [webpack](https://webpack.js.org/), which _requires_ a configuration file to work with `babel-loader`. Somewhat ironically, if you google "webpack react", [our webpack bundling page](https://reactjs.net/bundling/webpack.html) is the third result

We probably could add additional documentation for these three bundlers, but the getting started / tutorial should add as little friction as possible for new projects by recommending only one bundler.

Bonus points: This library should throw an error if attempting to use either `require` or `import` in server-side JS, and point to the bundler docs.

Fixes #816 

Major javascript bundlers (webpack, parcel) support emitting files with a content hash in the file path, along with an `index.html` or manifest containing references to these built files. Add a helper supporting referencing those files server-side for an improved developer experience around caching. Eg `/dist/my-app-1337.js`

Whenever a HTML helper is used to render `<script src>` tags, append some form of a hash to the emitted URL.

Another strategy would be to use a "cache-busting" query parameter that computes a hash from the Javascript source (since it's already in memory, we'd get this "for free"), but the actual file path would _not_ change. Eg `/dist/my-app.js?cb=1337`

See also: https://github.com/Daniel15/WebpackTag/
