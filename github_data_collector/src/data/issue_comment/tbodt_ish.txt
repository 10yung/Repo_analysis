 Space not freed on removal from iSH’s “Documents & Data” when content is copied out of iSH. This is on iOS 13.3.

Steps to reproduce:
1. Create a large file in iSH. I used `dd if=/dev/zero of=bigfile bs=10240 count=102400` for testing.
2. Go to iSH’s storage in the Settings app and record the size of the "Documents & Data".
3. Copy the large file from iSH via the Files app to another location local to the device (i.e. in On My iPad / On My iPhone).
4. Delete file in iSH (I used Files app to do this, but rm should also work)
5. Delete newly copied file and empty the trash
(Steps 4 and 5 can be reversed as well)
6. Go back to the main screen in the settings app and then go back into iSH’s storage (to refresh).

Space is not freed even though there are no references to the file. I think this may have to do with APFS cloning the file and not all references getting cleaned up.

Only workaround to free up space is to delete iSH and re-install. Rebooting doesn’t help.
iOS SIGKILLs an app if it goes into the background while holding a file lock. Every file operation can hold a file lock, so iSH can get killed fairly easily if you send it to the background while something is running. The only real solution is to request a 30 second background time extension and use it to hold up every task when it next gets to the end of handle_interrupt.
