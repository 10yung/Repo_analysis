As it turned out in #1134, some existing PHPBrew commands like `info` and `list-ini` are by design intended to be executed by the currently `use`d PHP, not the interpreter that runs PHPBrew.

The reasons to deprecate these commands are:
1. PHPBrew is meant to build and manage PHP. The fact that it runs on _a_ PHP is an implementation detail and should not be exposed via its commands.
2. The output of `phpbrew list-ini` is almost identical to the output of `php --ini`.
3. The output of `phpbrew info` is a subset of the output of `php -i`.

The currently used [corneltek/cliframework](https://packagist.org/packages/corneltek/cliframework) is unmaintained (https://github.com/c9s/CLIFramework/commit/5d7c4703390e80326938cbdef7de499bca2a0143) and unstable:

https://github.com/phpbrew/phpbrew/blob/0c75288ce0f67147fcae9216391b1d71d65d3b5a/composer.lock#L53-L54

The most obvious candidate for a replacement seems to be [symfony/console](https://packagist.org/packages/symfony/console) with its following benefits:
1. It's a de-facto standard for CLI applications in PHP according to [Packagist](https://packagist.org/?query=console).
2. It's well-maintained and -documented.
3. Supports dependency injection for commands.

Although, compared to the currently used framework, it also has downsides:
1. Most importantly, no OOTB support for shell completion (although there are 3rd-party solutions like [stecman/symfony-console-completion](https://packagist.org/packages/stecman/symfony-console-completion)). The completion solution needs to:
   1. Support Bash and Zsh and Fish.
   2. Note that we don't only need to be able to generate a static completion script but the script should also be able to call back to the application to complete the values that depend on the current state of the application (e.g. installed PHP versions, etc).
   3. We need to be able to specify predefined values for an argument like `phpbrew cd <var|etc|build|dist>`.
2. It's a bit opinionated about its defaults (e.g. the default `list` command needs to be reworked, the `--quet`, `--verbose` and similar arguments need to be implemented or removed, etc.).

An alternative solution might be to adopt the existing CLI framework with its all unmaintained dependencies and fix them but it seems to be a rabbit hole:

1. Version 4.0 seems to be significantly different from 3.0 (e.g. uses a pimple-based service container for wiring components).
2. Some of its dependencies ([universal/universal](https://packagist.org/packages/universal/universal) and [corneltek/codegen](https://packagist.org/packages/corneltek/codegen)) are needed for a handful of commands that PHPBrew doesn't need like `archive` and `github:build-topics`.

Maybe we could fork 3.0.x-dev, remove all unneeded dependencies and release a stable 3.0 under the PHPBrew account? But still, given how far behind the modern standards its codebase is, the maintenance burden alone seems to much.
