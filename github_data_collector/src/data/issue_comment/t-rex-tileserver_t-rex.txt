Bumps [http](https://github.com/hyperium/http) from 0.1.18 to 0.1.21. **This update includes security fixes.**
<details>
<summary>Vulnerabilities fixed</summary>
<p><em>Sourced from <a href="https://github.com/RustSec/advisory-db/blob/master/crates/http/RUSTSEC-2019-0034.toml">The RustSec Advisory Database</a>.</em></p>
<blockquote>
<p><strong>HeaderMap::Drain API is unsound</strong>
Affected versions of this crate incorrectly used raw pointer,
which introduced unsoundness in its public safe API.</p>
<p><a href="https://github-redirect.dependabot.com/hyperium/http/issues/354">Failing to drop the Drain struct causes double-free</a>,
and <a href="https://github-redirect.dependabot.com/hyperium/http/issues/355">it is possible to violate Rust's alias rule and cause data race with Drain's Iterator implementation</a>.</p>
<p>The flaw was corrected in 0.1.20 release of <code>http</code> crate.</p>
<p>Patched versions: &gt;= 0.1.20
Unaffected versions: none</p>
</blockquote>
<p><em>Sourced from <a href="https://github.com/RustSec/advisory-db/blob/master/crates/http/RUSTSEC-2019-0033.toml">The RustSec Advisory Database</a>.</em></p>
<blockquote>
<p><strong>Integer Overflow in HeaderMap::reserve() can cause Denial of Service</strong>
<code>HeaderMap::reserve()</code> used <code>usize::next_power_of_two()</code> to calculate the increased capacity.
However, <code>next_power_of_two()</code> silently overflows to 0 if given a sufficently large number
in release mode.</p>
<p>If the map was not empty when the overflow happens,
the library will invoke <code>self.grow(0)</code> and start infinite probing.
This allows an attacker who controls the argument to <code>reserve()</code>
to cause a potential denial of service (DoS).</p>
<p>The flaw was corrected in 0.1.20 release of <code>http</code> crate.</p>
<p>Patched versions: &gt;= 0.1.20
Unaffected versions: none</p>
</blockquote>
</details>

I'm trying to integrate t-rex-tileserver as backend to QGIS Desktop via the vector tiles reader Plugin available via http://plugins.qgis.org/plugins/vector_tiles_reader/.

When Connecting to t-rex, a URL pointing to style.json and tiles.json files is needed in the 
Vector Tiles Plugin. While connecting to t-rex that plugin sends HTTP HEAD requests to those URLs in order to check if they exist.

Unfortunately, t-rex answers with HTTP 404 when doing a HEAD reqeust on those URLS.
(see: https://github.com/geometalab/Vector-Tiles-Reader-QGIS-Plugin/blob/dev-qgis3/plugin/util/network_helper.py, line 12).

Which turns t-rex unreachable from that pluign.

Doing a hack, i patched VTR Plugin to do a classic HTTP GET, and further integration looks fine then, I can display vector tiles data inside QGIS.

Question: is it possible to implement HTTP HEAD for those tile.json and style.json Resources?
### Step1 
I download some data from [GADM](https://gadm.org/download_country_v3.html), and i import it to postgis using shp2pgsql.
``` sh
shp2pgsql -s 4326 -I ~/Downloads/gadm36_AFG_shp/gadm36_AFG_0.shp public.afg | psql -d gadm -U postgiser -W -h 0.0.0.0
```
So now I have a layer(afg).

### Step2
I transfrom the data to 3857 using the `st_transfrom(geom, 3857)`. 
``` sql
   select *  into public.afg_3857  
    from (select st_setsrid(st_transform(geom, 3857), 3857) as geom  
               from public.afg a) as foo;
```
So now, I have two layers(afg and afg_3857).

### Step3
I generate some point in afg polygon using the `st_generatepoints(geom, 2000)`.
``` sql
   select (st_dump(t.geom)).geom  
      into public.afg_point_4326  
     from (select st_generatepoints(geom, 2000) as geom  
                from public.afg a) as t;
```
So now I have three layers(afg, afg_3857 and afg_point_4326).

### Step4
I generate the config file using `t_rex genconfig`. Then I change afg_3857 srid to 3857 and afg_point_4326 to 4326(the original is 0, they cannot be auto recongized correctly).
